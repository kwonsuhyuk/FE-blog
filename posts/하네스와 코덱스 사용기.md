# 코덱스와 하네스를 써보며 느낀 AI 개발 도구의 역할

최근 개발을 하면서 AI 도구를 단순히 “코드를 대신 작성해주는 도구”가 아니라,  
개발 과정에서 반복되는 탐색과 검증 시간을 줄여주는 도구로 바라보게 되었다.

특히 코덱스와 하네스를 사용하면서 느낀 점은,  
AI가 개발자를 완전히 대체한다기보다 **문제를 더 빠르게 정리하고, 구현 방향을 비교하고, 반복 작업을 줄이는 데 도움을 준다**는 것이다.

다만 좋은 결과를 얻기 위해서는 AI에게 모든 판단을 맡기기보다,  
개발자가 문제의 맥락과 기준을 명확히 잡고 있어야 했다.

## 사용하게 된 배경

프론트엔드 개발을 하다 보면 코드를 작성하는 시간만큼이나  
기존 구조를 파악하고, 반복되는 코드를 정리하고, 구현 방향을 검토하는 데 많은 시간이 든다.

특히 프로젝트가 커질수록 단순히 “기능을 만든다”에서 끝나지 않는다.

- 기존 컴포넌트 구조와 충돌하지 않는지
- 상태 관리를 어디에서 처리하는 게 좋은지
- API 응답 데이터를 어떻게 화면에 맞게 가공할지
- 예외 상황에서는 어떤 흐름으로 처리할지
- 성능이나 유지보수 측면에서 문제가 생기지 않을지

이런 부분을 함께 고려해야 한다.

처음에는 이런 고민들을 모두 직접 정리하고 구현 방향을 찾는 데 시간이 오래 걸렸다.  
그러다 코덱스와 하네스 같은 AI 개발 도구를 사용해보면서,  
AI가 어느 부분에서 도움이 되고 어느 부분에서는 조심해야 하는지 직접 느낄 수 있었다.

## 처음 기대했던 점

처음에는 AI 도구를 사용하면 코드 작성 속도가 크게 빨라질 것이라고 기대했다.

예를 들어 컴포넌트를 만들거나, 타입을 정의하거나, 반복되는 유틸 함수를 작성할 때  
AI가 빠르게 초안을 만들어주면 개발 시간이 줄어들 것이라고 생각했다.

실제로 이런 부분에서는 도움이 됐다.

```ts
interface IUser {
  id: string;
  name: string;
  email: string;
  role: "admin" | "user";
}

interface IUserResponse {
  users: IUser[];
  totalCount: number;
}
```

위와 같은 단순 타입 정의나 API 응답 구조 정리는  
AI를 활용하면 빠르게 초안을 만들 수 있었다.

또한 반복되는 컴포넌트 구조를 잡거나,  
비슷한 패턴의 코드를 여러 곳에 적용해야 할 때도 속도 면에서 장점이 있었다.

하지만 사용하다 보니 중요한 건 코드 생성 속도 자체가 아니었다.

AI가 만들어준 코드가  
**현재 프로젝트 구조에 맞는지, 유지보수하기 좋은지, 실제 요구사항을 제대로 반영했는지**를 판단하는 과정이 더 중요했다.

## 실제로 느낀 문제

AI 도구를 사용하면서 가장 많이 느낀 문제는  
그럴듯하지만 프로젝트 맥락과 맞지 않는 코드가 나올 수 있다는 점이었다.

예를 들어 상태 관리를 컴포넌트 내부에서 처리해도 되는 상황인데  
불필요하게 전역 상태로 분리하거나, 반대로 여러 화면에서 공유되어야 하는 값을 지역 상태로만 처리하는 경우가 있었다.

또한 에러 처리나 예외 상황이 빠지는 경우도 있었다.

```ts
const fetchUser = async () => {
  const response = await fetch("/api/user");
  return response.json();
};
```

이 코드는 겉보기에는 동작할 것처럼 보인다.

하지만 실제 서비스 코드에서는 다음과 같은 부분을 함께 고려해야 한다.

- 요청 실패 시 어떻게 처리할 것인지
- 로딩 상태를 어디에서 관리할 것인지
- 인증이 만료되었을 때 어떻게 대응할 것인지
- 응답 데이터가 예상과 다를 때 화면이 깨지지 않는지
- 같은 요청이 반복될 때 캐싱은 어떻게 처리할 것인지

즉, AI가 코드를 만들어줄 수는 있지만  
그 코드가 서비스 코드로 들어가도 되는지는 결국 개발자가 판단해야 했다.

특히 프론트엔드에서는 화면이 정상적으로 보이는 것만으로 충분하지 않다.  
사용자가 어떤 흐름으로 기능을 사용하는지,  
데이터가 언제 바뀌고 화면에 어떻게 반영되는지,  
네트워크 지연이나 인증 만료 상황에서도 흐름이 끊기지 않는지까지 함께 봐야 한다.

## 어떻게 활용했는가

그래서 AI를 사용할 때 바로 코드를 맡기기보다,  
먼저 문제를 작게 나누는 방식으로 활용했다.

처음부터 “이 기능 만들어줘”라고 요청하기보다  
아래처럼 기준을 먼저 정리했다.

```md
- 현재 기능의 목적은 무엇인가
- 기존 구조에서 어디에 들어가야 하는가
- 상태는 지역 상태로 충분한가, 전역 상태가 필요한가
- 서버 상태는 TanStack Query로 관리하는 것이 적절한가
- API 실패나 로딩 상태는 어떻게 처리해야 하는가
- 사용자가 실제로 마주할 예외 상황은 무엇인가
```

이렇게 정리한 뒤 AI에게 질문하면 결과가 훨씬 나아졌다.

예를 들어 TanStack Query를 활용한 서버 상태 관리 구조를 고민할 때도  
단순히 “쿼리 코드 짜줘”라고 요청하기보다, 캐싱 기준과 실패 시 처리 방식을 함께 제시했다.

```ts
const { data, isLoading, isError } = useQuery({
  queryKey: ["user", userId],
  queryFn: () => getUser(userId),
  enabled: !!userId,
  staleTime: 1000 * 60,
});
```

이런 식으로 초안을 받은 뒤에는  
현재 프로젝트의 API 구조, 타입 정의, 에러 처리 방식에 맞춰 직접 수정했다.

또한 리팩토링을 할 때도 AI에게 바로 코드를 바꾸게 하기보다,  
먼저 현재 구조의 문제점을 정리하게 하고 여러 개선 방향을 비교했다.

예를 들어 컴포넌트가 너무 커졌을 때는 다음과 같은 식으로 질문했다.

```md
현재 컴포넌트가 UI 렌더링, API 요청, 상태 변경 로직을 모두 가지고 있다.
이 구조에서 어떤 기준으로 컴포넌트와 훅을 분리하는 것이 좋을지 비교해줘.
단, 기존 타입 구조와 TanStack Query 사용 방식은 유지하고 싶다.
```

이렇게 질문하면 단순히 코드를 새로 만들어주는 것이 아니라,  
어떤 기준으로 분리하면 좋은지 비교할 수 있었다.

결국 AI를 잘 쓰려면  
코드를 받는 것보다 **좋은 질문을 만드는 과정**이 더 중요했다.

## 코덱스와 하네스를 사용하며 느낀 차이

코덱스는 코드 작성과 수정 방향을 빠르게 잡는 데 도움이 됐다.

특정 함수나 컴포넌트의 초안을 만들거나,  
기존 코드의 개선 방향을 비교할 때 유용했다.

특히 반복되는 타입 정의, 유틸 함수 작성, 리팩토링 방향을 빠르게 검토할 때 속도감이 있었다.

반면 하네스는 단순히 코드를 생성하는 도구라기보다  
개발 흐름과 검증 과정을 함께 생각하게 해주는 도구에 가깝게 느껴졌다.

AI가 작성한 코드가 실제로 잘 동작하는지,  
배포나 테스트 흐름 안에서 어떻게 확인할 수 있는지,  
운영 환경에서는 어떤 문제가 생길 수 있는지를 함께 고려하게 됐다.

정리하면 이렇게 느꼈다.

| 도구 | 좋았던 점 | 주의할 점 |
| --- | --- | --- |
| 코덱스 | 코드 초안 작성, 리팩토링 방향 비교, 반복 코드 생성에 유용 | 프로젝트 맥락과 맞는지는 직접 검토해야 함 |
| 하네스 | 개발 흐름, 검증, 자동화 관점에서 생각하게 해줌 | 도구를 쓰는 목적과 적용 범위를 명확히 해야 함 |

두 도구 모두 생산성을 높여줄 수 있지만,  
무작정 적용하기보다는 “어떤 문제를 줄이기 위해 쓰는지”가 먼저 정리되어야 한다고 느꼈다.

## AI 도구를 사용할 때 중요하다고 느낀 기준

AI 개발 도구를 사용하면서 가장 크게 느낀 점은  
AI를 잘 쓰는 능력도 결국 개발 역량 위에 쌓인다는 것이다.

기본적인 코드 구조를 이해하지 못하면  
AI가 준 코드가 좋은 코드인지 판단하기 어렵다.

상태 관리, API 요청 흐름, 컴포넌트 분리 기준, 예외 처리 방식을 이해하고 있어야  
AI의 결과물을 그대로 쓰지 않고 프로젝트에 맞게 다듬을 수 있다.

내가 느낀 기준은 크게 세 가지였다.

### 1. AI에게 맡길 부분과 직접 판단할 부분을 구분해야 한다

AI에게 맡기기 좋은 부분은 비교적 명확했다.

- 반복되는 타입 정의
- 단순 유틸 함수 초안
- 컴포넌트 구조 초안
- 리팩토링 방향 비교
- 코드 리뷰 관점에서의 문제점 탐색

반대로 직접 판단해야 하는 부분도 있었다.

- 현재 프로젝트 구조와 맞는지
- 팀의 코드 스타일과 맞는지
- 실제 사용자 흐름을 해치지 않는지
- 예외 상황을 충분히 고려했는지
- 유지보수하기 쉬운 구조인지

AI가 빠르게 답을 줄 수는 있지만,  
그 답을 선택할 기준은 개발자가 가지고 있어야 했다.

### 2. 요구사항을 구체적으로 전달할수록 결과가 좋아진다

AI에게 모호하게 요청하면 결과도 모호하게 나온다.

예를 들어 다음과 같은 요청은 결과가 흔들리기 쉽다.

```md
이 컴포넌트 리팩토링해줘.
```

반면 기준을 함께 주면 결과가 훨씬 나아진다.

```md
이 컴포넌트에서 API 요청 로직은 커스텀 훅으로 분리하고,
UI 컴포넌트는 props 기반으로 재사용 가능하게 만들고 싶다.
기존 TypeScript 타입은 유지하고, TanStack Query 구조도 유지해줘.
```

결국 AI에게 좋은 결과를 얻기 위해서는  
내가 먼저 문제를 잘 이해하고 있어야 했다.

### 3. 결과물을 그대로 믿지 않고 검증해야 한다

AI가 작성한 코드는 그럴듯해 보일 때가 많다.  
하지만 그럴듯한 코드와 실제로 서비스에 넣을 수 있는 코드는 다르다.

그래서 AI가 만든 코드는 항상 직접 확인했다.

- 타입 에러가 없는지
- 런타임 에러 가능성은 없는지
- 예외 상황이 빠지지 않았는지
- 기존 코드 스타일과 맞는지
- 실제 화면에서 사용자 흐름이 자연스러운지

특히 인증, 결제, 데이터 수정처럼 사용자 영향이 큰 기능에서는  
AI가 만든 코드를 그대로 쓰기보다 흐름을 다시 검증하는 과정이 필요했다.

## 마무리

코덱스와 하네스를 사용해보면서  
AI 개발 도구를 바라보는 기준이 조금 바뀌었다.

처음에는 코드를 빠르게 만들어주는 도구라고 생각했지만,  
지금은 **개발자가 더 좋은 판단을 하기 위해 사용하는 보조 도구**에 가깝다고 느낀다.

AI가 만들어준 결과물을 그대로 사용하는 것이 아니라,  
문제를 구조화하고, 초안을 빠르게 만들고, 더 나은 방향을 비교하는 데 활용할 때 가장 효과적이었다.

결국 중요한 것은 AI를 얼마나 많이 쓰느냐가 아니라,  
AI가 만든 결과를 내 프로젝트의 맥락에 맞게 판단하고 수정할 수 있는가라고 생각한다.

앞으로도 AI 도구를 사용할 때 단순히 빠르게 코드를 만드는 데 그치지 않고,  
더 나은 구조를 고민하고 검증하는 방향으로 활용해보고 싶다.

## 한 줄 정리

AI는 개발을 대신해주는 도구라기보다,  
개발자가 더 빠르게 고민하고 더 정확하게 판단할 수 있도록 돕는 도구에 가깝다.
