인턴 업무를 진행할때 정말 많이 들었던 용어가 **비즈니스 로직과 도메인 로직** 이다.

특히 FSD 아키텍처를 고민하고 공부할때 비즈니스 로직을 분리해야 한다는 말을 정말 많이 들었었는데 지금 생각해보면 정확하게 알지 못하고 어렴풋하게 알고 애매하게 용어를 사용하고 있었던 것 같다. 그래서 이번기회에 정확하게 해당 용어가 무엇을 뜻하고 어떤 곳에서 사용하는 로직인지 알아보려고 한다.



## 비즈니스 로직이란?

비즈니스 로직은 서비스가 해결하려는 "문제"와 관련된 규칙과 흐름을 의미한다.

예를 들어,

- 회원가입 시 이메일 중복 검사
- 결제 시 할인 정책 적용
- 출퇴근 기록 처리

처럼 서비스의 정책과 흐름을 담당하는 로직이다.

그렇다면, 

## 도메인 로직이란?

도메인 로직은 특정 "도메인(문제 영역)" 자체의 규칙을 의미한다.

예를 들어 근태 관리 서비스라면,

- 출근 시간 계산
- 근무 시간 산정
- 휴가 일수 계산

처럼 도메인 자체의 본질적인 규칙을 처리하는 로직이다.

## 비즈니스 로직 vs 도메인 로직

| 구분      | 비즈니스 로직          | 도메인 로직              |
| --------- | ---------------------- | ------------------------ |
| 목적      | 서비스 정책 처리       | 도메인의 핵심 규칙 처리  |
| 관심사    | "어떻게 동작할 것인가" | "무엇이 맞는 값인가"     |
| 예시      | 할인 적용, 권한 체크   | 근무시간 계산, 점수 계산 |
| 변경 빈도 | 자주 바뀜 (기획 영향)  | 상대적으로 안정적        |



한줄로 정리하면, 

- **비즈니스 로직은 "서비스 흐름",**
- **도메인 로직은 "데이터의 진짜 규칙"** 



## 실제 예시를 통한 이해해보기

> 휴가 신청 기능

근태 관리 서비스에서 "휴가 신청" 기능이 있다고 가정해보자.

이 기능을 담당하는 코드를 뜯어보면 다음과 같은 작업들이 있다.

- 사용자의 남은 휴가 일수를 확인한다.
- 신청하려는 휴가 기간이 유효한지 검증한다.
- 유효하지 않으면 에러 메시지를 표시한다.
- 승인 상태를 "대기"로 설정한다.
- 관리자에게 알림을 전송한다.
- 휴가 데이터를 DB에 저장한다.

이 코드를 도메인 로직과 비즈니스 로직으로 나누어보면,

**도메인 로직**은 "휴가"라는 문제 자체에 대한 의사결정을 담당한다.

- 남은 휴가 일수가 충분한지 확인
  → 휴가 신청이 가능한지 판단

- 휴가 기간이 유효한지 검증
  → 시작일과 종료일이 올바른지 판단

- 승인 상태를 "대기"로 설정
  → 휴가 신청의 상태 흐름 정의

이 로직들은 "휴가 도메인"의 핵심 규칙을 담당한다.

**비즈니스 로직**은 도메인 로직이 동작할 수 있도록 환경을 구성하고,
결과를 외부로 전달하는 역할을 한다.

- 유효하지 않으면 에러 메시지 표시
  → UI 처리

- 관리자에게 알림 전송
  → 외부 시스템과의 통신

- 휴가 데이터를 DB에 저장
  → Persistence 처리

이 로직들은 도메인 로직의 결과를 사용자나 시스템에 반영하는 역할을 한다.



## 코드 예시

```tsx
// 도메인 로직
function validateVacation(daysLeft: number, requestDays: number) {
  if (daysLeft < requestDays) {
    throw new Error("휴가 일수가 부족합니다.");
  }
}

function createVacationStatus() {
  return "PENDING";
}
```

```tsx
// 서비스 로직
async function requestVacation(userId: string, days: number) {
  const daysLeft = await getUserVacationDays(userId);

  validateVacation(daysLeft, days); // 도메인 로직

  const status = createVacationStatus(); // 도메인 로직

  await saveVacation(userId, days, status); // DB 저장

  await sendNotificationToAdmin(userId); // 외부 시스템
}
```



결론적으로,

**도메인 로직은 -> 판단하는 로직**

**비즈니스 로직은 -> 연결하는 로직** 이다. 



## 왜 이걸 구분해야 할까?

도메인 로직과 비즈니스 로직을 구분하는 이유는 단순히 개념을 나누기 위함이 아니라,
코드의 유지보수성과 확장성을 높이기 위함이다.

비즈니스 로직과 도메인 로직이 섞여 있게 되면

- 정책 변경 시 영향 범위를 파악하기 어렵고
- 여러 곳에서 동일한 로직이 중복되며
- 테스트가 어려워지는 문제가 발생한다

반대로 도메인 로직을 분리하면

- 핵심 규칙을 한 곳에서 관리할 수 있고
- 다양한 UI나 API에서 재사용이 가능하며
- 변경에도 유연하게 대응할 수 있다

실제로 인턴 당시 권한 검증 로직이 여러 파일에 흩어져 있어
정책이 바뀔 때마다 오류가 반복되는 문제가 있었다.
이후 로직을 분리하고 구조를 정리하면서
변경에 대한 대응 속도와 안정성이 크게 개선되었다.

결국 중요한 것은
"이 로직이 무엇을 판단하는가"를 기준으로 나누는 것이다.

앞으로는 기능을 구현할 때
이 로직이 도메인 로직인지, 아니면 단순히 흐름을 제어하는 비즈니스 로직인지
한 번 더 고민해보는 습관을 가져야겠다고 생각했다.
