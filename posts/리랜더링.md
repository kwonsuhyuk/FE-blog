# React에서 불필요한 리렌더링 줄이기

React 프로젝트를 진행하다 보면 처음에는 기능 구현에 집중하게 된다.

버튼을 클릭하면 상태가 바뀌고, API 데이터를 받아와 화면에 보여주고, 사용자가 입력한 값을 저장하는 흐름을 만드는 것이 우선이다.

하지만 프로젝트 규모가 커질수록 단순히 기능이 동작하는 것만으로는 부족하다는 것을 느꼈다.  
화면에 표시되는 데이터가 많아지고, 여러 컴포넌트가 같은 상태를 바라보게 되면 작은 상태 변경에도 예상보다 많은 컴포넌트가 다시 렌더링될 수 있기 때문이다.

이번 글에서는 React에서 불필요한 리렌더링을 줄이기 위해 어떤 부분을 확인해야 하는지 정리해보려고 한다.

## 리렌더링은 무조건 나쁜 것이 아니다

먼저 리렌더링 자체가 무조건 나쁜 것은 아니다.

React는 상태나 props가 변경되면 컴포넌트를 다시 렌더링한다.  
이 과정에서 변경된 UI를 계산하고 실제 화면에 반영한다.

문제는 필요한 컴포넌트만 다시 렌더링되는 것이 아니라, 변경과 직접 관련 없는 컴포넌트까지 함께 렌더링되는 경우다.

예를 들어 부모 컴포넌트의 상태 하나가 변경됐을 뿐인데, 하위에 있는 리스트, 차트, 모달 컴포넌트까지 모두 다시 렌더링된다면 성능에 영향을 줄 수 있다.

특히 데이터가 많은 테이블이나 리스트에서는 이런 문제가 더 크게 느껴질 수 있다.

## 상태 위치 확인하기

리렌더링 문제를 볼 때 가장 먼저 확인한 것은 상태의 위치였다.

상태가 너무 상위 컴포넌트에 있으면 작은 상태 변경에도 하위 컴포넌트 전체가 영향을 받을 수 있다.

예를 들어 검색 input의 값만 변경되는 상황을 생각해볼 수 있다.

```tsx
function EmployeePage() {
  const [keyword, setKeyword] = useState("");

  return (
    <>
      <SearchInput value={keyword} onChange={setKeyword} />
      <EmployeeSummary />
      <EmployeeTable />
    </>
  );
}
```

이 구조에서는 `keyword`가 변경될 때마다 `EmployeePage`가 다시 렌더링된다.  
그 결과 `keyword`와 직접 관련이 없는 `EmployeeSummary`까지 함께 렌더링될 수 있다.

검색어가 `SearchInput`과 `EmployeeTable`에서만 필요하다면 상태를 더 좁은 범위로 옮기거나, 관련 컴포넌트만 영향을 받도록 구조를 분리하는 것이 좋다.

상태는 무조건 상위에 두는 것이 아니라, 실제로 필요한 위치에 두는 것이 중요하다.

## 컴포넌트 역할 분리하기

하나의 컴포넌트가 너무 많은 역할을 가지고 있으면 상태 변경의 영향 범위도 커진다.

예를 들어 페이지 컴포넌트 하나에서 API 호출, 필터 상태, 테이블 렌더링, 모달 제어까지 모두 처리하면 작은 변경에도 페이지 전체가 영향을 받을 수 있다.

```tsx
function EmployeePage() {
  return (
    <>
      <EmployeeFilter />
      <EmployeeTable />
      <EmployeeDetailModal />
    </>
  );
}
```

이렇게 필터, 테이블, 모달을 역할별로 분리하면 각 컴포넌트가 담당하는 책임이 명확해진다.

컴포넌트를 분리한다고 해서 항상 성능이 좋아지는 것은 아니지만, 어떤 상태가 어떤 컴포넌트에 영향을 주는지 파악하기 쉬워진다.  
결과적으로 리렌더링 문제를 찾고 개선하기도 쉬워진다.

## 전역 상태는 필요한 값만 구독하기

Zustand 같은 전역 상태 관리 라이브러리를 사용할 때도 주의할 점이 있다.

store 전체를 가져오면 내가 사용하지 않는 값이 변경되어도 컴포넌트가 다시 렌더링될 수 있다.

```tsx
const userStore = useUserStore();
```

이 방식보다는 컴포넌트에서 필요한 값만 선택해서 가져오는 것이 좋다.

```tsx
const userName = useUserStore((state) => state.userName);
```

상태 변경 함수도 필요한 것만 가져온다.

```tsx
const setUserName = useUserStore((state) => state.setUserName);
```

전역 상태를 사용하는 것 자체보다 중요한 것은 컴포넌트가 어떤 상태를 구독하고 있는지다.

필요한 값만 구독하면 상태 변경의 영향 범위를 줄일 수 있고, 컴포넌트가 더 예측 가능하게 동작한다.

## useMemo와 useCallback은 필요한 곳에만 사용하기

리렌더링 최적화와 관련해서 `useMemo`와 `useCallback`이 자주 사용된다.

하지만 이 둘을 무조건 많이 사용하는 것이 좋은 것은 아니다.

`useMemo`는 계산 비용이 큰 값을 메모이제이션할 때 유용하다.

```tsx
const filteredEmployees = useMemo(() => {
  return employees.filter((employee) =>
    employee.name.includes(keyword)
  );
}, [employees, keyword]);
```

직원 목록이 많고 검색어나 필터가 변경될 때마다 계산이 반복된다면 `useMemo`를 사용할 수 있다.

`useCallback`은 함수를 props로 넘길 때 유용하다.

```tsx
const handleSelectEmployee = useCallback((id: string) => {
  setSelectedEmployeeId(id);
}, []);
```

특히 자식 컴포넌트가 `React.memo`로 감싸져 있는 경우, 매 렌더링마다 새로운 함수가 생성되는 것을 줄이는 데 도움이 된다.

다만 단순한 계산이나 가벼운 컴포넌트에 무조건 적용하면 코드만 복잡해질 수 있다.  
그래서 실제로 렌더링 비용이 있거나, props 참조값 때문에 불필요한 렌더링이 발생하는 경우에 사용하는 것이 좋다.

## React.memo 적용 기준

`React.memo`는 props가 변경되지 않았다면 컴포넌트를 다시 렌더링하지 않도록 도와준다.

```tsx
interface IEmployeeItemProps {
  name: string;
  status: string;
}

const EmployeeItem = React.memo(function EmployeeItem({
  name,
  status,
}: IEmployeeItemProps) {
  return (
    <li>
      {name} - {status}
    </li>
  );
});
```

리스트 아이템처럼 같은 구조가 반복되고, props가 자주 바뀌지 않는 컴포넌트에는 효과적일 수 있다.

하지만 모든 컴포넌트에 `React.memo`를 적용하는 것은 좋은 방식이 아니다.

props가 매번 새 객체나 새 함수로 전달된다면 memo의 효과가 줄어든다.  
또한 컴포넌트 자체가 가볍다면 비교 비용이 더 클 수도 있다.

그래서 `React.memo`는 반복 렌더링이 많거나, 실제 렌더링 비용이 있는 컴포넌트 위주로 적용하는 것이 좋다고 생각한다.

## 서버 상태와 클라이언트 상태 분리하기

리렌더링을 줄이기 위해서는 서버 상태와 클라이언트 상태를 구분하는 것도 중요하다.

API로 가져오는 데이터까지 모두 전역 상태에 넣으면 데이터 갱신, 로딩, 에러, 캐싱을 직접 관리해야 한다.

서버에서 가져오는 데이터는 TanStack Query 같은 도구로 관리하고, Zustand는 모달 상태, 선택된 탭, 사용자 UI 흐름 같은 클라이언트 상태를 관리하는 방식이 더 적합하다.

```tsx
const { data, isLoading } = useQuery({
  queryKey: ["employees"],
  queryFn: getEmployees,
});
```

이렇게 역할을 분리하면 상태의 책임이 명확해진다.  
또한 불필요하게 전역 상태가 커지는 것을 막을 수 있다.

## 정리

React에서 리렌더링은 자연스러운 동작이다.  
중요한 것은 리렌더링 자체를 없애는 것이 아니라, 불필요한 리렌더링이 발생하지 않도록 구조를 설계하는 것이다.

정리하면 다음과 같다.

1. 상태는 실제로 필요한 위치에 둔다.
2. 컴포넌트는 역할별로 분리한다.
3. 전역 상태는 필요한 값만 구독한다.
4. `useMemo`, `useCallback`, `React.memo`는 필요한 곳에만 사용한다.
5. 서버 상태와 클라이언트 상태를 구분한다.

성능 최적화는 특정 Hook을 많이 사용하는 것보다 상태와 컴포넌트의 책임을 명확히 나누는 것에서 시작된다고 생각한다.
