지금까지 여러 프로젝트를 진행하면서 가장 많이 한 고민중에 하나가 프로젝트의 기능을 추가하거나 수정할때, 각각의 파일의 의존성이 꼬이고 이 파일은 어디에 두어야 할지 항상 고민하면서 프로젝트를 진행했던 경험이 있다. 그 과정에서 같이 협업하는 팀원과의 소통에도 많은 비용이 필요한 문제점이 있었다.

이런 문제점을 해결할 방법을 항상 고민하고 있었고, 그에 따라 확장성 및 유지보수성을 높이기 위해 여러 프로젝트에 FSD 아키텍처를 도입하고 적용한 경험을 공유할 예정이다.

### FSD 아키텍처란 무엇인가?
공식문서에 따르면 FSD 아키텍처는 Feature-Sliced Design(기능 분할 설계) 의 약자로, 프론트엔드 애플리케이션 구조를 위한 아키텍처 방법론이다.

![Image](https://github.com/user-attachments/assets/1f1d4968-83a3-4f5e-9eb4-4979f6cd2654)

출처: FSD 공식문서

FSD 에서의 핵심은 코드를 3가지 차원으로 분리하여 재사용 가능한 로직을 통제하고, 높은 응집도와 낮은 결합도를 유지하는 것이다. <br/>
3가지 차원은 수직적인 레이어(Layers), 수평적인 슬라이스(Slices), 그리고 기술적 목적에 따른 세그먼트(Segments) 로 나누어 관리하는 것이다.

> Layers (계층): 수직적 구조

계층은 프로젝트의 가장 최상위 수준이며, **아래에 있는 계층은 위의 계층을 절대 참조할 수 없다**는 엄격한 규칙이 적용

| 계층 (Layer) | 설명 | 주요 포함 요소 |
| :--- | :--- | :--- |
| **`app`** | 애플리케이션의 입구입니다. 전역 설정이 위치합니다. | Providers, Router, Global Styles |
| **`processes`** | 여러 페이지에 걸친 복잡한 흐름을 관리합니다. (최근 생략 추세) | 결제 프로세스, 인증 워크플로우 |
| **`pages`** | 실제 라우팅의 주체입니다. 여러 위젯을 조합합니다. | Page Components |
| **`widgets`** | 독립적인 기능 단위를 가집니다. 가장 큰 조립 단위입니다. | Header, Sidebar, ProductCardGrid |
| **`features`** | **사용자에게 가치를 주는 기능** 단위입니다. | AddToWishlist, LoginByPhone |
| **`entities`** | 비즈니스 모델과 실체입니다. 데이터와 기본 UI를 다룹니다. | User, Product, Order (Schema & State) |
| **`shared`** | 특정 도메인에 얽매이지 않는 공통 요소입니다. | UI Kit, API Client, Utils, Constants |

* **단방향 의존성:** `features`는 `entities`를 가져다 쓸 수 있지만, 반대는 불가능하다.
* **추상화 수준:** 위로 올라갈수록(app 방향) 실행 로직에 가깝고, 아래로 내려갈수록(shared 방향) 범용적인 도구에 가깝다.

> Slices (슬라이스): 수평적 분리

shared와 app 계층을 제외한 나머지 계층은 비즈니스 도메인에 따라 슬라이스로 나뉘는데. 이는 관련 있는 코드끼리 응집도를 높이는 역할을 한다. <br/>
**특징으로는, 같은 계층 내의 슬라이스끼리는 서로를 직접 참조할 수 없다.**

- 예시 (entities 계층 내부):

  `entities/user`

  `entities/product`

  `entities/cart`

> Segments (세그먼트): 내부 구성 요소

각 슬라이스는 목적에 따라 더 작은 세그먼트로 쪼개진다. 파일의 역할이 명확해져 유지보수가 쉬워지며, 팀원 간에 "어떤 코드가 어디에 있을지"에 대한 예측 가능성을 높여주는 역할을 한다.

| 세그먼트 (Segment) | 설명 | 주요 포함 요소 |
| :--- | :--- | :--- |
| **`ui`** | 사용자에게 보여지는 화면 요소를 정의합니다. | 리액트 컴포넌트, 스타일 파일 (CSS/SCSS/Styled-components) |
| **`model`** | 해당 슬라이스의 데이터 구조와 상태 변화를 다룹니다. | 상태 관리(Zustand, Redux), 비즈니스 로직, 데이터 변환(Mapper) |
| **`api`** | 외부 서버와의 통신을 담당합니다. | 서버 요청 로직 (Axios instance, React Query Hooks, API 함수) |
| **`lib`** | 슬라이스 내부 로직을 지원하는 유틸리티입니다. | 해당 슬라이스 전용 보조 함수(Helper), 커스텀 훅 |


- **Public API 준수:** 각 슬라이스의 최상위 `index.ts`에서 필요한 `ui`, `model`, `api`만 export 하여 외부에서 내부 세그먼트 구조를 직접 참조하지 않도록 한다.
- **선택적 사용:** 모든 슬라이스에 모든 세그먼트가 있을 필요는 없다. 기능에 따라 `ui`만 있거나 `api`만 있을 수도 있다.

### FSD로 얻는 이점 'Public API'와 '단방향 의존성'


FSD가 단순한 폴더 정리를 넘어 '아키텍처'로 기능하게 만드는 핵심 규칙입니다.

| 핵심 개념 | 설명 | 효과 |
| :--- | :--- | :--- |
| **Public API (`index.ts`)** | 각 슬라이스와 세그먼트의 입구입니다. 내부의 복잡한 구조를 숨기고 필요한 요소만 외부로 노출(export)합니다. | **캡슐화:** 내부 코드를 리팩토링해도 외부 코드(import 하는 곳)를 수정할 필요가 없습니다. |
| **단방향 의존성** | 상위 계층(`pages`, `widgets`)은 하위 계층(`features`, `entities`)을 참조할 수 있지만, 반대는 절대 불가능합니다. | **선명함:** 스파게티처럼 꼬인 의존성을 방지하고, 코드 흐름을 예측 가능하게 만듭니다. |

해당 두 이점을 통해서 필요한 코드만 외부로 노출하고, 코드흐름을 예측가능하게 하여서 유지보수를 용이하게 하고, 명확한 프로젝트 흐름을 구성할 수 있는 이점이 있다.

---

### 실전 도입 시 마주치는 고민들

이론은 완벽하지만, 실제 프로젝트에 적용하다 보면 반드시 마주치는 질문들이 몇개 있다.

#### 1. 모든 계층을 다 쓰는 게 정답일까? 
처음에는 모든 계층을 최대한 다 사용하기 위해서 노력하면서 설계를 진행했었다. 하지만, 모든 계층을 다 사용하는 것은 오히려 프로젝트를 진행하는 것에 있어 더 많은 고민을 하게 되었고, 처음 사용하는 것에 대해서 진입장벽이 매우 높았다. 

따라서, 설계중에 필요하지 않은 계층을 과감히 사용하지 않고, 최대한 사용하는 계층만을 사용하도록 설계하여 FSD 를 사용하는 이점을 최대한 가져갈 수 있도록 노력했다 .

우선, **processes 계층의 삭제**: 유저의 복잡한 시나리오보다는 단발성 CRUD(생성, 조회, 수정, 삭제)가 주를 이루는 프로젝트에서 processes는 과도한 추상화였다. 따라서, 과감히 삭제하고 로직을 features나 pages로 분산시켰다. (또한 최근에는 processes 계층을 생략하는 추세라고 한다.. )

또한, 처음에는 features만으로 충분할 줄 알았지만, 여러 도메인이 섞인 '대시보드 카드'나 '복합 필터바'를 처리할 때 widgets 계층이 없으면 의존성이 꼬이기 시작했다.
따라서, widgets 를 사용해서 의존성이 꼬이는 문제를 좀 더 명확한 구조로 해결하였다. 그 외에도, shared 를 사용해서 공통적인 요소 처리등을 하여, 좀 더 유지보수가 용이한 프로젝트 구조를 정립했다.
결론적으로, FSD는 **선택적 도입**이 가능하다. 팀의 상황에 맞게 app - pages - widgets - features - entities - shared 중 필요한 것만 골라 쓰는 유연함이 필요했다.


#### 2. 이건 기능(Feature)인가, 데이터(Entity)인가 ?
해당 고민이 아마 FSD 를 설계하면서 가장 많이 하고 여러웠던 부분 중 하나이다. <br/>
단순히 '데이터냐 기능이냐'로 나누기에는 실제 비즈니스 로직이 얽혀 있는 회색지대가 너무 많았기 때문이다.

이 혼란을 해결하기 위해 세운 핵심 판단 기준은 **비즈니스 가치의 능동성**이었다. (비즈니스 로직 가치에 대해서는 다음 포스트에.. )

> Entity vs Feature: 수동적 존재와 능동적 액션의 차이

해당 고민을 해결하기 위해서 컴포넌트가 사용자에게 주는 **가치**의 성격에 따라 계층을 분리했다.

- Entity (비즈니스 실체): 데이터 그 자체와 이를 시각적으로 표현하는 **수동적**인 컴포넌트다. (예: 유저 프로필 카드, 상품 상세 정보 레이아웃) 이는 데이터 구조(Schema)에 종속되며, 그 자체로 어떤 상태 변화를 일으키지 않는다.

- Feature (비즈니스 기능): 데이터를 조작하거나 특정 액션을 유도하는 **능동적**인 컴포넌트이다. (예: 유저 권한 변경 버튼, 재고 수정 폼, 검색 필터링) 사용자의 의도가 반영되어 시스템에 '변화'를 주는 로직이 포함된다.


> 그렇다면 "유저 상세 페이지의 '수정' 버튼은 어디로 가야 하나?"

실제로 백오피스 기능을 구현하며 마주친 가장 큰 난제는 **데이터와 액션이 결합된 작은 단위들**이었다. 예를 들어 유저 상세 정보 하단에 붙은 '상태 변경(활성/비활성) 스위치'는 과연 Entity일까, Feature일까? 이 작은 스위치 하나를 두고도 많은 의견이 갈렸다.

FSD에는 이를 칼같이 나누는 절대적인 정답이 없었기에, 우리는 이 문제를 해결하기 위해 **독립적 재사용성**이라는 우리만의 기준을 세워 합의해 나갔다.

1. Feature로 분류한 경우: "어디서든 꽂으면 돌아가는 독립 부품" <br/>
해당 수정 로직(API 호출, 유효성 검사, 성공/실패 토스트 알림 등)이 유저 상세 페이지뿐만 아니라 유저 목록 리스트, 혹은 대시보드 등 다른 곳에서도 동일하게 사용될 가능성이 높다면 이를 독립적인 feature로 보았다.

- 의도: 이렇게 분리하면 어떤 페이지나 위젯에서든 해당 기능을 가져다 꽂기만 하면 동일하게 동작한다. 비즈니스 로직이 한곳에 응집되어 있어, 추후 기획이 변경되어도 해당 feature 폴더 내부만 수정하면 된다는 확신을 가질 수 있었다.

2. Entity 내부의 UI로 정의한 경우: "데이터 구조에 묶인 일체형 부품"<br/>
반면, 해당 수정 액션이 오직 그 특정 데이터 구조 안에서만 유효하고, 다른 곳에서 재사용될 여지가 전혀 없다고 판단되면 entities/--/ui 내부에 포함시켰다.

- 의도: 모든 작은 버튼이나 스위치를 기계적으로 feature로 쪼개면 파일 경로가 지나치게 깊어지고 관리 포인트만 늘어나는 오버 엔지니어링이 발생했다. 재사용성이 없는 단순 액션은 entity 내부로 밀어 넣음으로써, 불필요한 추상화 비용을 줄이고 파일 시스템의 복잡도를 팀이 감당 가능한 수준으로 유지했다.

프로젝트에 FSD를 도입하며 깨달은 가장 큰 사실은, FSD가 모든 상황에 답을 내려주는 마법의 규칙이 아니라는 점이었다. 오히려 모호한 지점이 생길 때마다 동료들과 모여 
 이 코드는 여기에 있는 게 우리 팀의 생산성에 도움이 될까? 를 논의하게 만드는 언어인 것 같았다.

#### 3. shared 계층의 비대화

또한 FSD 를 사용하면서 생긴 문제는 shared 계층이 무한 증식한다는 점이었다. 어디에 넣기 애매한 공통 컴포넌트, 유틸 함수, 타입 정의들을 하나둘씩 shared에 던져 넣다 보니, 어느새 이 폴더는 예전의 쓰레기통 같던 components 폴더가 되어가고 있는 문제가 생겼다.  

이를 최대한 방지하기 위해서, shared 내부에서도 ui, api, lib 세그먼트를 엄격히 나누어 관리했다. 하지만 가장 효과적이었던 규칙은 **도메인 냄새가 나면 무조건 상향**하는 것이었다. 조금이라도 특정 비즈니스 도메인(예: 유저, 상품)의 성격을 띠고 있다면, 설령 재사용성이 높더라도 과감히 entities로 옮겨 도메인 간 경계를 명확히 했다.


이러한 고민을 통해서 기준이 확립되자, <br/>
더 이상 "이 코드는 어디에 넣지?"라는 소모적인 고민 대신 "이 로직의 비즈니스 가치는 무엇인가?"라는 본질적인 설계 고민에 집중할 수 있었다.



### 마치며

FSD 아키텍처 도입은 작업자들이 처음에 학습이 필요하다는 부담이 있었지만, 점진적 도입이 가능하고 장기적인 관점에서 일관성, 확장성, 그리고 유지보수성을 크게 향상시키는 효과를 가져올 수 있었다. 또한, 장기적인 확장 가능한 서비스 구조를 구축을 통해서, 곧 개발 생산성을 높이는 핵심 동력이 될 수 있었던 것 같다. 
해당 아키텍처 공부를 통해서 실전 프로젝트에 적용해본 경험을 통해서 앞으로도 기능 추가, 수정에 대해서 좀 더 빠르게 적용할 수 있는 기반을 다진 것 같고, 좋은 전략적 도구가 된것 같다.
